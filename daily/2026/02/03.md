### Stored Cross-Site Scripting (XSS) in Rich Text Editor

| **Symbol**         | `useEffect.arg1`                                                                                                            |
| **Tags**           | `_trace_scan_generic`, `deduplicated`                                                                                       |

#### Description

The `JsonFormsProseControl` component is vulnerable to stored Cross-Site Scripting (XSS). The component uses a `useEffect` hook to programmatically set the content of a Tiptap rich text editor from the `data` prop. This `data`, which holds user-saved content, is passed directly to the `editor.commands.setContent()` method without prior sanitization. The Tiptap editor, as configured in the application, does not sanitize content passed to this method. According to the application's privilege model, the 'Editor' role is attacker-assumable and is permitted to create and modify content. This allows a malicious user with 'Editor' privileges to inject arbitrary HTML, including script tags or event handlers, into a rich text field. When a privileged user, such as an 'Admin' or 'Publisher', subsequently opens this content in the editor, the `useEffect` hook executes `setContent` with the malicious payload. This causes the script to execute in the privileged user's browser, leading to potential session hijacking, unauthorized actions, or other malicious activities. The presence of `// eslint-disable-next-line @typescript-eslint/no-unsafe-argument` suggests awareness of a type-safety issue, but the underlying XSS vulnerability was not addressed.

#### Affected Code

```typescript
useEffect(() => {
  if (data !== undefined) {
    const selection = editor?.state.selection;
    if (!selection) return;
    // eslint-disable-next-line @typescript-eslint/no-unsafe-argument
    editor.commands.setContent(data);
    editor.commands.setTextSelection(selection);
  }
}, [data]);
```

#### LLM Judge Assessments

When a higher-privileged user, such as an 'Admin' or 'Publisher', reviews this content, the `setContent` function executes the payload, leading to a stored cross-site scripting vulnerability.

_PoC:_ An attacker with 'Editor' privileges can save a rich text field with a payload like `<img src=x onerror=alert(document.cookie)>`. When an 'Admin' or 'Publisher' user opens this content in the editor, the script will execute, potentially leading to session hijacking.

#### Description

The file upload mechanism allows an authenticated user (e.g., an Editor) to control the `Content-Type` of an uploaded file. The `handleUpload` function in `useUploadAssetMutation.ts` uses the `file.type` property from the client-side `File` object directly as the `Content-Type` header when uploading to the presigned URL. The backend does not enforce any `Content-Type` restrictions when generating the presigned URL. Additionally, the `Content-Disposition` is set to `inline`. An attacker can upload a file containing a malicious HTML/SVG payload (e.g., JavaScript), bypass client-side checks by giving it an allowed extension (like `.jpg`), but intercept the upload request to set the `Content-Type` to `text/html` or `image/svg+xml`. When another user views this uploaded asset, the browser will render it as HTML/SVG and execute the embedded script in the context of the asset-serving domain. If this domain is not properly sandboxed, this leads to Stored XSS, which can be used to steal session tokens, perform actions on behalf of the user, or escalate privileges.

#### Affected Code

```typescript
const handleUpload = async ({ file, presignedPutUrl }: HandleUploadParams) => {
  // ...
  const response = await fetch(presignedPutUrl, {
    headers: {
      "Content-Type": file.type, // User-controlled Content-Type
      "Content-Disposition": `inline; filename*=UTF-8''${encodedFilename}`, // Inline disposition
    },
    method: "PUT",
    body: file,
  });
  // ...
};
```

4. Initiate the upload of `malicious.jpg`.
5. Intercept the `PUT` request to the S3 presigned URL.

7. Allow the request to complete.

### Regular Expression Denial of Service (ReDoS) in reference link processing

| Field              | Value                                                                                             |
| ------------------ | ------------------------------------------------------------------------------------------------- |
| **Severity**       | High                                                                                              |
| **Confidence**     | 0.9                                                                                               |
| **False Positive** | No                                                                                                |
| **Validated By**   | agent                                                                                             |
| **Created By**     | trace_scan:generic                                                                                |
| **File Path**      | `./isomer/apps/studio/src/features/editing-experience/utils/getHtmlWithRelativeReferenceLinks.ts` |
| **Symbol**         | `getHtmlWithRelativeReferenceLinks`                                                               |
| **Tags**           | `_trace_scan_generic`, `security`, `validation`                                                   |

#### Description

The function `getHtmlWithRelativeReferenceLinks` in [`isomer/apps/studio/src/features/editing-experience/utils/getHtmlWithRelativeReferenceLinks.ts:6-15`](./isomer/apps/studio/src/features/editing-experience/utils/getHtmlWithRelativeReferenceLinks.ts:6-15) uses a regular expression that is vulnerable to Regular Expression Denial of Service (ReDoS). The pattern includes the sub-expression `[^[]*\/`, where the greedy quantifier `*` in `[^[]*` can match the `/` character, which is also explicitly matched by the subsequent `\/`. This ambiguity forces the regex engine to backtrack extensively when processing a string with many `/` characters that does not ultimately match the full pattern. An attacker with content editing privileges (e.g., an `Editor`) can submit crafted HTML containing a long path-like string in an `href` attribute. This input is processed on the server, causing the Node.js event loop to block due to excessive CPU consumption, leading to a denial of service for the entire application.

#### Affected Code

```typescript
export const getHtmlWithRelativeReferenceLinks = (html: string) =>
  html.replaceAll(
    new RegExp(
      `href="(?:http|https):

*\/${REFERENCE_LINK_REGEX.source}"`,
      "g",
    ),
    (_, siteId, resourceId) => {
      return `href="[resource:${siteId}:${resourceId}]"`;
    },
  );
```

#### Proof of Concept

```python
import re
import time

# This PoC demonstrates the ReDoS vulnerability using Python's `re` module,
# which has a similar regex engine to JavaScript's.

# The vulnerable part of the regex is `[^[]*\/`.
# We assume REFERENCE_LINK_REGEX.source is `\\[resource:\\w+:\\w+\\]` based on context.
# The effective regex is: 'href="(?:http|https)://[^[]*/\\[resource:(\\w+):(\\w+)\\]'
regex = r'href="(?:http|https)://[^[]*/\\[resource:(\w+):(\w+)\\]'

def check_redos(input_str):
    print(f"Testing input: '{input_str[:40]}...' ({len(input_str)} chars)")
    start_time = time.time()
    try:
        re.search(regex, input_str)
    except Exception as e:
        print(f"An error occurred: {e}")
    end_time = time.time()
    duration = end_time - start_time
    print(f"Execution time: {duration:.6f} seconds")
    return duration

# A benign string that matches quickly.
benign_input = '<html><a href="https://example.com/a/[resource:1:2]">link</a></html>'

# A malicious string designed to cause catastrophic backtracking.
# It has many '/' characters that `[^[]*` can match, followed by a character
# that prevents the rest of the regex from matching, forcing backtracking.
malicious_input = '<html><a href="https://' + 'a/' * 25 + 'b[resource:1:2]">link</a></html>'

# A slightly more malicious string to show the quadratic complexity.
more_malicious_input = '<html><a href="https://' + 'a/' * 30 + 'b[resource:1:2]">link</a></html>'

print("--- Testing ReDoS Vulnerability ---")
check_redos(benign_input)
print("-" * 20)
check_redos(malicious_input)
print("-" * 20)
check_redos(more_malicious_input)
print("\nNote the non-linear increase in execution time, which indicates a ReDoS vulnerability.")
```

#### LLM Judge Assessments

**Judge 1:** Confidence: `high` | False Positive: `False` | Severity: `high`

> The regular expression contains a pattern `[^[]*\/` which is vulnerable to Regular Expression Denial of Service (ReDoS). The `[^[]*` is greedy and can match the `/` character, which is then explicitly matched by `\/`, leading to catastrophic backtracking on crafted inputs. According to the documentation, the "Editor" role is attacker-assumable and can modify content.
